03Deployment models

Three ways to cut infrastructure costs.

We're not selling a better MP3 — that already exists. We offer real architectural flexibility: three deployment modes that replace today's 5-6 streaming bitrate versions with 1-2 well-defined copies.

A.

Server-side reconstruction

Store only the compressed version. Your servers reconstruct at delivery time. The client receives ready-to-play audio, with no changes to its current flow.

~50-70 % less storage
Server CPU on smaller files
B.

Device-side reconstruction

The user's phone decompresses locally. The server becomes a byte-delivery service. Compute load distributes massively across millions of devices.

~50-70 % less storage
Server CPU near zero
C.

Two strategic copies

One bit-perfect version for archival and audiophiles, one max version for mass streaming. You replace today's 5+ bitrates with just 2 copies, served by context.

Less transcoding
Fewer decisions, lower costs

05Frequently asked

What you're going to ask.

Why deliver an APK and not source code?

Because source code isn't the product. The product is the compression system and its protections operating together. Sharing isolated source code without the protected binary would eliminate the layers that justify the licensing model.

We don't protect the algorithm by hiding it: we protect it by making copying economically irrational. Delivering the APK is itself a demonstration of confidence in the product's protections.

How do I know the benchmark isn't hardcoded?

Because the benchmark runs live, on your file, on your device. The app doesn't carry fixed numbers: it executes the real codec against the WAV you choose and reports what it measures. The expected vs obtained MD5 hash is computed on screen.

Additionally, you can delete the original WAV after compressing and play the resulting .tfm: if it works, it's because the decompression is real, not a pointer to the original.

Does it compete with FLAC?

It competes in lossless compression on certain audio profiles (structured, voice, podcast) where TRANSFORMER can match or exceed FLAC. On dense musical audio, FLAC remains competitive.

The central difference isn't just the ratio: TRANSFORMER offers three architectural deployment models (server/client/dual), integrated cryptographic verification, and proprietary anti-reverse-engineering protections that FLAC, being open, cannot offer to a commercial platform.

What if someone decompiles it?

The binary incorporates two independent obfuscation layers (design-level mathematical + Flutter's native AOT compilation) and an active protection system called Adaptive Integrity Shield that detects analysis environments and silently alters codec behavior.

Combined, we estimate an asymmetric cost ratio of approximately 150:1 between construction and functional reverse engineering. It's defense, not secrecy: retains value even when fully known.

What file sizes does the demo support?

The demonstration build is optimized for interactive evaluation with files from 1 to 20 MB. Functional range extends to ~30 MB. Significantly larger files may exhaust available memory — this is expected behavior for a reference implementation without streaming.

Migration to the C++ engine with FFI (Q4 2026) introduces block-based processing and removes this limit. The .tfm format is identical in both versions; only the memory strategy changes.

How much does licensing cost?

The licensing model depends on the use case: volume, integration, regional exclusivity, required support. We don't publish prices — they are negotiated by contract per partner's terms.

If your case fits one of the three architectural models (A, B, C), reach out for an exploratory 30-minute call. It's the best way to understand whether it makes sense for both parties.
